## ✅ Applied (in this PR)
- **Duplicate booking runs cause bookings_found>0 but sms_sent=0 — seen_ids prevents re-SMS but re-scrapes** (`agent.py`, impact: low) — The run history shows a clear recurring pattern: the FIRST run of each day has bookings_found=1, sms_sent=1, but every subsequent run that same day shows bookings_found=1, sms_sent=0 (e.g. 2026-07-11 10:39 sends 1, then 11:43, 12:54, 14:18, 15:28, 16:25, 17:28 all find 1 but send 0). This is because the booking stays on the Nya tab (Förfrågan/Offert have no Hanterad button, so they are never cleared), and each run re-scrapes it but skips SMS because it is in seen_ids. That is correct behavior for dedup, but the PENDING Förfrågan/Offert warning is logged every run without any escalation. This is expected but should at least be surfaced in the run summary so the owner knows a booking needs manual portal action.
- **Förfrågan bookings never trigger parts_agent as documented** (`agent.py`, impact: high) — SKILL.md and agent.py docstring state that Förfrågan bookings should launch parts_agent.py to build a quote. However, in agent.py's run() loop there is NO call to parts_agent for booking_type == 'Förfrågan'. The booking only gets an SMS and a PENDING warning. This is a documented behavior that is missing entirely from the code. Since parts_agent.py is reported as FILE NOT FOUND, wiring it in now would break, so this must not be auto-applied — but the gap should be flagged.
- **mark_booking_handled opens a second full browser+login per booking — slow and login not verified** (`portal_scraper.py`, impact: medium) — mark_booking_handled() launches a fresh browser and re-logs in for EVERY confirmed Bokning, doubling browser open time (SKILL.md priority 5: reduce browser open time). More importantly, _fill_login()'s return value is ignored here (unlike fetch_bookings which checks it), so a silent login failure would proceed to click Hanterad on a login page and return False without a clear reason. Add a login-success check.
- **qty extraction assumes column index 3 — fragile across tables** (`portal_scraper.py`, impact: medium) — _extract_specification reads qty = cells[3].inner_text() when len(cells) > 3. Different tables (services vs parts vs fluids) may not have qty in the 4th column, producing wrong 'x N' suffixes in the SMS (SKILL.md priority 2: SMS content quality). Guard by only treating the cell as qty if it is a small integer.
